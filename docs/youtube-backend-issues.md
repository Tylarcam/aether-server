# YouTube Backend: Issues, Attempts & Current State

## The Core Problem

YouTube actively blocks server-side video access from cloud provider IPs (AWS, GCP, Modal).
Any request from a datacenter IP — regardless of yt-dlp version or flags — is treated as bot
traffic and rejected with a sign-in wall. This is not a yt-dlp bug; it is intentional YouTube
anti-scraping policy enforced at the network level.

---

## Attempt Log

### Attempt 1 — Mobile player clients (ios / android / tv_embedded)
**Goal:** Bypass age-gate by impersonating a mobile client, which historically skips age verification.

**Change:**
```python
--extractor-args "youtube:player_client=ios"
```
Fallback loop: ios → android → tv_embedded

**Result:** ✅ Fixed age-gate for some videos. ❌ Introduced new error on other videos.

**New error:**
```
ERROR: [youtube] k-ITV61q3B0: Requested format is not available.
```

**Root cause:** Mobile clients expose a different format manifest than the web client.
`--audio-format mp3` requires a specific audio stream that often doesn't exist in the
mobile manifest. The fallback loop also only retried on "Sign in"/"age" errors, so a
format error caused a hard fail on the first client without trying the rest.

---

### Attempt 2 — Flexible format selection
**Goal:** Let yt-dlp pick whatever audio stream is available instead of forcing MP3.

**Change:**
```python
# Before
"--extract-audio",
"--audio-format", "mp3",
"--audio-quality", "64K",
"--max-filesize", "24m",

# After
"--extract-audio",
"--format", "bestaudio/best",
```

Also expanded the retry condition to include format errors:
```python
is_retryable = (
    "Sign in" in stderr or
    "age" in stderr.lower() or
    "format is not available" in stderr.lower() or
    "requested format" in stderr.lower()
)
```

Also added `None` (default web client) as first in the loop: `(None, "ios", "android", "tv_embedded")`

**Result:** ✅ Format error gone. ❌ Web + all mobile clients now return:
```
YouTube is blocking server-side access to this video.
```
Even Pharrell's "Happy" (ZbZSe6N_BXs) — a fully public, non-restricted video — fails.

**Root cause:** YouTube has escalated bot detection. Modal server IPs are flagged regardless
of client type. The sign-in wall is shown to all datacenter IPs, not just for restricted content.

---

### Attempt 3 — Update yt-dlp image
**Goal:** Force Modal to rebuild with a fresh yt-dlp (was cached on an old version).

**Change:**
```python
# Before
.pip_install("fastapi", "groq", "pydantic", "yt-dlp")

# After
.pip_install("fastapi", "groq", "pydantic", "yt-dlp[default]")
```

`yt-dlp[default]` adds: brotli, pycryptodomex, websockets, yt-dlp-ejs — all optional
dependencies that improve YouTube extraction reliability. New version: 2026.3.17.

**Result:** ❌ Still blocked. YouTube's IP-level block predates yt-dlp version.

---

### Attempt 4 — YouTube session cookie
**Goal:** Authenticate server-side requests using the user's real YouTube session.

**File provided:** `session_id.md` contained:
```
session_logininfo=AFmmF2swRAIgL1n_E5T_...
```

**Change:** Stored as Modal secret `youtube-session`, passed to yt-dlp:
```python
--add-header "Cookie: session_logininfo=VALUE"
```

**Result:** ❌ Still blocked.

**Root cause:** YouTube authentication requires a full cookie set — at minimum: `SID`, `HSID`,
`SSID`, `APISID`, `SAPISID`, `__Secure-1PAPISID`, `__Secure-3PAPISID`. The `session_logininfo`
cookie is a login-state indicator, not an auth token. Passing it alone does not satisfy YouTube's
authentication check. A full cookie export (Netscape format via a browser extension like
"Get cookies.txt LOCALLY") would be needed — but those cookies expire and rotate, making
them impractical to maintain in a secret.

## What Would Actually Work (Not Yet Implemented)

### Option A — PO Token (Proof of Origin)
yt-dlp 2024.11+ supports `--extractor-args "youtube:po_token=web+BASE64"`. A PO token is
generated by running JavaScript in a browser context. Projects like `bgutil-ytdlp-pot-provider`
can generate them automatically, but require a running Node.js service alongside the Python backend.

**Complexity:** High. Requires a sidecar service and token rotation.

### Option B — Full cookie export from the extension
Use `chrome.cookies.getAll({ domain: '.youtube.com' })` to collect ALL YouTube cookies from the
user's browser, pass them to Modal as a Netscape-format cookies.txt, and provide them to yt-dlp
via `--cookies`. This uses the user's real authenticated session.

**Complexity:** Medium. Requires the `"cookies"` permission in the manifest (visible to users
in the Chrome Web Store listing). Cookie values would be transmitted to Modal — a privacy
consideration for a public extension.

### Option C — Residential proxy
Route yt-dlp requests through a residential IP (Bright Data, Oxylabs, etc.). YouTube does not
block residential IPs.

**Complexity:** Low code change, but has ongoing cost and terms-of-service implications.

## Current Solution (Implemented)

The Modal yt-dlp path is retained as a fallback but is effectively non-functional for YouTube.
The primary YouTube transcription path uses the **user's own browser session**:

1. When a YouTube URL is submitted, Aether checks for an open tab with that video.
2. If no tab is open, Aether opens a background tab silently (user stays on their current page).
3. The background tab loads with the user's YouTube session (signed in, age-verified).
4. A script injected into `window` reads `ytInitialPlayerResponse.captions`, fetches the
   timed-text JSON from within the page context (carries session cookies automatically),
   and returns the parsed transcript.
5. The background tab is closed immediately after extraction.

**What it covers:**
- All public videos with captions (auto-generated or manual)
- Age-restricted videos (user must be signed in to YouTube in their browser)
- Any video the user can personally watch

**What it does NOT cover:**
- Videos with no captions and no auto-generated captions (typically: music videos, old content,
  non-English content without auto-captions enabled)
- Live streams

**For caption-less videos:** User is directed to use the Record tab — play the video in YouTube,
use Aether's tab capture feature to record the audio, and transcribe the recording through the
working Modal + Groq audio endpoint.
